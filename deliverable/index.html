Phase 5: Kubernetes Deployment

  • Kubernetes manifests for all services:
    All manifests are organized under k8s/ in the orchestrating repo, one folder per environment.
    Kubernetes Manifests in Orchestrating Repo

    The structure is:
    • k8s/namespaces.yaml — defines dev, staging, prod namespaces
    • k8s/dev/, k8s/staging/, k8s/prod/ — one folder per environment, each containing:
      • configmap.yaml — non-sensitive config (DB_HOST, DB_NAME)
      • secret.yaml — sensitive data (DB_USER, DB_PASSWORD) base64-encoded
      • resource-quota.yaml — resource caps per namespace
      • db-storage.yaml — PersistentVolume + PersistentVolumeClaim for PostgreSQL
      • database.yaml — Deployment + ClusterIP Service
      • products.yaml — Deployment + ClusterIP Service + NodePort Service
      • orders.yaml — Deployment + ClusterIP Service + NodePort Service
      • frontend.yaml — Deployment + NodePort Service



  • Running pods in different namespaces:
    Three namespaces were created — dev, staging, and prod — each with their own isolated set of pods. Each environment uses the image tag that Jenkins pushes for that branch (:dev from develop, :staging from release/*, :latest from main). Prod runs 2 replicas for frontend, products, and orders to demonstrate high availability.

  • Services and configuration:
    Each service exposes two Kubernetes Services:
    • ClusterIP — for internal pod-to-pod communication (e.g. orders calls http://product-service:3001)
    • NodePort — for external access from the host machine
    The database only has a ClusterIP since it should never be exposed externally.

  • Persistent volumes for the database:
    Each environment has its own PersistentVolume (cluster-scoped, using Minikube hostPath) and PersistentVolumeClaim (namespace-scoped). The database pod mounts the PVC at /var/lib/postgresql so data survives pod restarts and redeployments. All PVs use Retain reclaim policy so data is never lost on PVC deletion.

    EnvironmentPV NameHost PathCapacity
    devdb-pv-dev/mnt/data/ecommerce-dev1Gi
    stagingdb-pv-staging/mnt/data/ecommerce-staging2Gi
    proddb-pv-prod/mnt/data/ecommerce-prod5Gi

    PersistentVolumes (kubectl get pv):

  • Resource quotas per namespace:
    Each namespace has a ResourceQuota to prevent one environment from consuming all cluster resources.


  • Deployment strategy:
    All Deployments use Rolling Update with maxUnavailable: 0 and maxSurge: 1. This means Kubernetes always spins up a new pod and waits for it to pass its readiness probe before terminating the old one — guaranteeing zero downtime during image updates triggered by Jenkins.

    Each service has readiness and liveness probes:
    • Database: pg_isready command
    • Application services: HTTP GET on /health
    ConfigMaps hold non-sensitive configuration (DB_HOST, DB_NAME) and Secrets hold sensitive data (DB_USER, DB_PASSWORD), so neither are hardcoded in the Deployment specs.
